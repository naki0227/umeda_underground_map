# Global Development Rules

## 目的

このルールは、AIにコーディングを依頼するときに、短期的に動くコードではなく、長期的に保守・拡張しやすいコードを作らせるためのものです。

AIは単なる実装者ではなく、設計補助者、レビュー者、実装者として振る舞うこと。

実装速度よりも、設計の妥当性、テスト容易性、保守性、拡張性、CIによる品質保証を優先する。

---

## 最重要方針

以下の順序を必ず守る。

1. CIの有無を確認する
2. CIがなければ先に作る
3. 要件を整理する
4. 影響範囲を確認する
5. 技術スタックと採用理由を説明する
6. 長期的なメリット・デメリットを説明する
7. 規模に合ったアーキテクチャを決める
8. テスト方針を決める
9. コミット分割方針を決める
10. 最小差分で実装する
11. type check / format / lint / test / build を確認する
12. 変更内容と残課題を報告する

CIがない状態で機能実装を始めてはいけない。

---

## Definition of Done

作業は、実装が終わっただけでは完了ではない。

以下を満たして初めて完了とする。

- 要件を満たしている
- CIがすべて通っている
- unit testが追加・更新されている
- 必要に応じてintegration testが追加・更新されている
- type check / format / lint / test / build が通っている
- 既存機能を壊していない
- エラーハンドリングが実装されている
- 必要なログが適切に入っている
- セキュリティ上の懸念が確認されている
- DB変更がある場合はmigrationとrollback方針がある
- API変更がある場合はrequest / response / status codeが明確である
- 変更内容が説明できる
- 残課題が明記されている
- 小さなコミットに分けられる状態になっている

テスト、lint、format、typecheck、buildを実行できていない場合は、完了扱いしてはいけない。

実行できない場合は、その理由と代替の確認方法を明記する。

---

## CI First

新規プロジェクトまたはCIが未整備のプロジェクトでは、機能実装より先に必ずCIを作る。

CIが存在しない状態で、アプリケーション機能の実装を開始してはいけない。

CIでは最低限、以下を必ず実行する。

- type check
- formatter check
- lint
- unit test
- build check

可能であれば、以下も追加する。

- integration test
- dependency check
- security check
- test coverage report

CIで落ちる状態のまま完了扱いしてはいけない。

CIを実行できない場合は、その理由とローカルで代替実行したコマンドを明記する。

---

## CIがない場合の作業順序

CIがないプロジェクトでは、作業順序を必ず以下にする。

1. パッケージマネージャを確認する
2. ビルドツールを確認する
3. formatter設定を追加する
4. lint設定を追加する
5. type check設定を追加する
6. unit test設定を追加する
7. build check設定を追加する
8. GitHub ActionsなどのCI設定を追加する
9. CIが通る最小状態を確認する
10. その後に機能実装を開始する

CI整備と機能実装を同じコミットに混ぜない。

---

## 実装前に必ず行うこと

実装前に、必ず以下を短く整理する。

1. CIの有無
2. CIで実行される検証内容
3. 要件の確認
4. 影響範囲
5. 採用する技術スタック
6. その技術を使う理由
7. 長期的に見たメリット
8. 長期的に見たデメリット
9. ディレクトリ構成
10. 責務分離
11. テスト方針
12. コミット分割方針

不明点がある場合でも、明らかに推測できる場合は最小限の仮定を置いて進める。

大きな設計判断が必要な場合は、実装前に選択肢と推奨案を提示する。

---

## 技術選定ルール

技術スタックを選ぶときは、流行や実装の楽さだけで選ばない。

必ず以下の観点で説明する。

- 開発速度
- 保守性
- 型安全性
- テスト容易性
- チーム開発しやすさ
- 将来的な拡張性
- パフォーマンス
- エコシステム
- 学習コスト
- 運用コスト

既存プロジェクトでは、明確な理由なく技術スタックを変更しない。

新しい依存ライブラリを追加する場合は、以下を説明する。

- 追加理由
- 代替案
- 追加しない場合の選択肢
- 長期的な影響
- メンテナンスリスク

---

## ADR 方針

重要な技術選定や設計判断を行う場合は、ADRを作成する。

ADRは Architecture Decision Record の略であり、なぜその判断をしたのかを後から追えるようにするための記録である。

ADRには以下を含める。

- 背景
- 課題
- 選択肢
- 採用した案
- 採用理由
- メリット
- デメリット
- 将来的な見直し条件

ADRを作成すべき対象例。

- フレームワーク選定
- DB選定
- 認証方式
- 状態管理
- ディレクトリ構成
- アーキテクチャ方針
- 外部サービス導入
- CI/CD方針
- ログ・監視方針

小さな実装判断までADRにしすぎない。

長期的に影響する判断、後から理由を説明する必要がある判断だけを記録する。

---

## セキュリティ方針

実装時は常にセキュリティを考慮する。

以下を禁止する。

- 秘密情報をコードに直書きする
- API key / token / password をログ出力する
- 認証なしで重要なAPIを公開する
- 認可チェックを省略する
- SQL injectionにつながる文字列結合
- XSSにつながる危険なHTML挿入
- CSRF対策なしの状態変更処理
- CORSを安易に全許可する
- エラー詳細をそのままユーザーに返す
- 個人情報を不要に保存する
- 個人情報を不要にログへ出力する

実装時は以下を確認する。

- 認証が必要な処理か
- 認可が必要な処理か
- 入力値検証があるか
- 秘密情報の扱いは安全か
- ログに機密情報が出ていないか
- 外部入力を信用していないか
- エラーメッセージから内部情報が漏れていないか
- 依存ライブラリに既知の脆弱性がないか

セキュリティを後回しにしない。

特に認証、認可、入力値検証、秘密情報管理は設計段階で考慮する。

---

## ログ・監視・エラー設計

本番運用を想定して、必要な箇所にログを入れる。

ログでは以下を意識する。

- 障害調査に必要な情報を残す
- 個人情報や秘密情報を出力しない
- エラー発生箇所と原因が追えるようにする
- 正常系で過剰にログを出しすぎない
- request id / user id など追跡に必要な情報を扱う
- 外部APIやDBアクセス失敗時に原因を追えるようにする

エラーは以下を区別する。

- ユーザー入力エラー
- 認証エラー
- 認可エラー
- 外部APIエラー
- DBエラー
- 想定外の内部エラー

ユーザーに返すエラーメッセージと、内部ログに残すエラー詳細は分ける。

ユーザーには必要最小限の情報を返し、内部ログには調査に必要な情報を残す。

ただし、内部ログにも秘密情報や個人情報を出力してはいけない。

---

## アーキテクチャ方針

プロジェクト規模に応じて、言語・フレームワークに合ったクリーンアーキテクチャを採用する。

ただし、過剰設計は禁止する。

小規模な機能に対して不要に層を増やさない。

設計では以下を優先する。

- 責務分離
- テスト容易性
- 変更容易性
- 依存方向の明確化
- 外部サービスやフレームワークへの依存の隔離
- 長期的な保守性

---

## 小規模プロジェクトの方針

小規模プロジェクトでは、以下を守る。

- ルーティング、UI、ロジックを最低限分離する
- 1ファイルに責務を詰め込まない
- 将来的に分割しやすい構造にする
- テストしやすい関数単位に分ける
- 外部APIやDBアクセスを直接あちこちに書かない

小規模であっても、責務分離とテスト容易性は妥協しない。

ただし、小規模な機能に対して過度な抽象化や不要なレイヤーを増やしてはいけない。

---

## 中規模プロジェクトの方針

中規模プロジェクトでは、以下を意識する。

- domain
- application
- infrastructure
- presentation

ビジネスロジックをUIやhandlerに書かない。

外部API、DB、UIフレームワークへの依存を内側に漏らさない。

usecase単位で責務を分ける。

テストでは、domainとapplicationを優先的に単体テストする。

---

## 大規模プロジェクトの方針

大規模プロジェクトでは、Clean Architecture または Hexagonal Architecture を基本にする。

以下を守る。

- domainを中心に置く
- usecase単位で責務を分ける
- repository interfaceを定義する
- infrastructureは差し替え可能にする
- presentation層を薄く保つ
- 外部サービスへの依存を境界で止める
- テストしやすい依存方向にする

依存方向は外側から内側に向ける。

domain層がframework、DB、HTTP、外部APIに依存してはいけない。

---

## ファイルサイズ

1ファイルは原則300行以内にする。

300行を超えそうな場合は、責務を見直して分割する。

ただし、以下は例外として許容する。

- 型定義だけのファイル
- 自動生成ファイル
- テストケースが多いファイル
- 明確に分割すると逆に可読性が下がる場合

例外にする場合は理由を説明する。

---

## テスト方針

ユニットテストは必須。

実装だけを追加してテストを追加しないことは禁止する。

テストできない場合は、その理由と代替の確認方法を説明する。

テストでは以下を意識する。

- 正常系
- 異常系
- 境界値
- 空データ
- 不正入力
- 権限・認証が関係する場合の失敗ケース
- DBや外部APIに依存しないロジックの単体テスト

バグ修正では、先に再現テストを追加してから修正する。

テストは実装の後付けではなく、設計の一部として扱う。

---

## フロントエンド実装ルール

フロントエンドでは、機能単位で責務を分ける。

### コンポーネント

- 1コンポーネント1責務を原則とする
- UI表示とビジネスロジックを混ぜすぎない
- propsの型を明示する
- 巨大なコンポーネントを作らない
- 再利用可能なUIはcomponentsに分離する
- ページ固有のUIはfeaturesまたはpages配下に置く
- 表示だけのコンポーネントと状態を持つコンポーネントを分ける

### hooks

- 1hookは1つの関心ごとに限定する
- API通信、状態管理、UI制御を無理に1つのhookにまとめない
- hookには必要に応じてユニットテストを用意する
- 副作用の範囲を明確にする
- hooksの命名は責務が分かるものにする

### TypeScript

- any は原則禁止
- APIレスポンスには型を定義する
- 外部入力は信用しない
- 型で防げるバグは型で防ぐ
- 無理な型アサーションをしない
- unknownを使う場合は適切にnarrowingする
- 型定義と実装の責務を混ぜすぎない

---

## API設計ルール

APIは一貫性を重視する。

APIを追加・変更する場合は、以下を明確にする。

- endpointの責務
- HTTP method
- request schema
- response schema
- status code
- error response
- pagination
- filtering
- sorting
- authentication
- authorization

API responseはフロントエンド都合だけで決めない。

将来的な拡張性と後方互換性を考慮する。

破壊的変更が必要な場合は、versioningや段階的移行を検討する。

同じ意味のstatus codeやerror responseを機能ごとにバラバラにしない。

APIの入力値は必ず検証する。

---

## バックエンド実装ルール

バックエンドでは、ドメイン、ユースケース、インフラ、API層を分離する。

### model / domain

modelを追加・変更する場合は、必ずユニットテストを含める。

modelには以下を含める。

- ドメイン上の制約
- 値のバリデーション
- 状態遷移
- ビジネスルール

modelにDBやHTTPの都合を混ぜない。

modelだけを追加してtestがないコミットは禁止する。

### usecase / service

usecaseはアプリケーション固有の処理を書く。

HTTP requestやDB queryを直接扱わない。

usecaseを追加・変更する場合は、正常系・異常系のテストを追加する。

usecaseだけを追加してtestがないコミットも原則禁止する。

### repository

repositoryは永続化の抽象化を担当する。

domainやusecaseが具体的なDB実装に依存しないようにする。

DB実装を追加する場合は、可能な範囲でintegration testを用意する。

### handler / controller

handlerは薄く保つ。

handlerの責務は以下に限定する。

- requestの受け取り
- inputの変換
- usecaseの呼び出し
- responseへの変換
- status codeの決定

ビジネスロジックをhandlerに書かない。

---

## DB変更ルール

DB schemaを変更する場合は、必ずmigrationを作成する。

DB変更では以下を確認する。

- 既存データへの影響
- rollback方法
- null制約の影響
- indexの必要性
- unique制約の必要性
- 外部キー制約の必要性
- パフォーマンス影響
- 本番データ量で安全に実行できるか

破壊的変更をいきなり行わない。

カラム削除や型変更が必要な場合は、段階的migrationを検討する。

DB変更とアプリケーションコード変更のデプロイ順序を意識する。

既存データがある前提で安全に変更する。

---

## Rustルール

Rustプロジェクトでは、以下を守る。

- unwrap / expect は原則禁止
- domain層はaxumやsqlxに依存しない
- usecase層はHTTPを知らない
- repositoryはtraitで抽象化する
- DB実装はinfrastructureに置く
- handlerは薄く保つ
- errorは型で表現する
- modelには必ずunit testを書く
- usecaseにはmock repositoryを使ったunit testを書く
- エラーを文字列だけで雑に扱わない
- ResultとOptionを適切に使う
- cloneの乱用を避ける

RustのCIには最低限以下を含める。

- cargo fmt check
- cargo clippy
- cargo test
- cargo check
- cargo build

---

## TypeScript / React / Next.jsルール

TypeScript、React、Next.jsプロジェクトでは、以下を守る。

- any は原則禁止
- 1コンポーネント1責務
- 1hook1責務
- API通信はコンポーネントに直接書かない
- schema validationを導入できる場合は導入する
- server stateとUI stateを混ぜない
- component / hook / util には必要に応じてtestを書く
- propsの型を明示する
- 巨大なpage componentを作らない
- UIとビジネスロジックを分離する

TypeScriptのCIには最低限以下を含める。

- typecheck
- format check
- lint
- test
- build

---

## Goルール

Goプロジェクトでは、以下を守る。

- packageの責務を明確にする
- handlerにビジネスロジックを書かない
- interfaceは使う側に置く
- errorを握りつぶさない
- contextを適切に扱う
- table driven testを活用する
- go fmtを必ず通す
- go vetを必ず通す

GoのCIには最低限以下を含める。

- go fmt check
- go vet
- go test
- go test race
- go build

---

## Ruby / Railsルール

RubyまたはRailsプロジェクトでは、以下を守る。

- modelに過剰な責務を持たせない
- service objectを適切に使う
- controllerを薄く保つ
- validationを明確にする
- RSpecを使う場合は責務ごとにspecを分ける
- model spec、request spec、service specを適切に使い分ける
- RuboCopを通す
- security checkを可能な限り導入する

Ruby / RailsのCIには最低限以下を含める。

- rubocop
- rspec
- rails test
- brakeman
- bundle audit

---

## Java / Spring Bootルール

JavaまたはSpring Bootプロジェクトでは、以下を守る。

- controllerを薄く保つ
- serviceにアプリケーションロジックを集約する
- entityにDB都合以外の責務を詰め込みすぎない
- DTOを適切に分ける
- repositoryを直接controllerから呼ばない
- unit testとintegration testを分ける
- nullを雑に扱わない
- 例外設計を明確にする

JavaのCIには最低限以下を含める。

- test
- check
- build
- formatter check
- lint check

---

## Pull Request 方針

PRは1つの目的に限定する。

1つのPRに以下を混ぜない。

- 新機能
- 大規模リファクタ
- UI変更
- DB設計変更
- 依存ライブラリ更新
- CI変更
- フォーマットだけの変更

PRには以下を含める。

- 目的
- 変更内容
- 影響範囲
- テスト内容
- 動作確認方法
- スクリーンショットまたはAPI実行例
- リスク
- ロールバック方法

レビューしやすい単位に分ける。

巨大なPRを作らない。

---

## コミット単位

コミットは小さく分ける。

複数の責務を1コミットに混ぜない。

CI整備、設計変更、機能追加、テスト追加、リファクタリングは可能な限り分ける。

---

## フロントエンドのコミット単位

フロントエンドでは、以下のような単位でコミットする。

- 1コンポーネント
- 1hook
- 1画面
- 1つのUI状態
- 1つのAPI接続
- 1つのバリデーション追加
- 1つのテスト追加

複数のコンポーネントやhookをまとめて変更する場合は、理由を説明する。

---

## バックエンドのコミット単位

バックエンドでは、以下のような単位でコミットする。

- model追加 + model test
- usecase追加 + usecase test
- repository interface追加
- repository実装 + repository test
- handler追加 + handler test
- migration追加
- 認証処理追加
- エラーハンドリング追加

modelだけを追加してtestがないコミットは禁止する。

usecaseだけを追加してtestがないコミットも原則禁止する。

---

## コミットメッセージ

コミットメッセージは変更内容が分かるように書く。

推奨形式は以下。

- feat(frontend): add task card component
- feat(frontend): add useTaskList hook
- feat(backend): add task model with tests
- feat(backend): add create task usecase
- test(backend): add task validation tests
- refactor(frontend): split task form state hook
- fix(backend): validate empty task title
- ci: add typecheck, lint, format and test workflow

---

## リファクタリング方針

リファクタリングは目的を明確にして行う。

以下を守る。

- 振る舞いを変えない
- 変更前後でテストが通ることを確認する
- 機能追加と大規模リファクタを同じPRに混ぜない
- 命名変更だけの変更とロジック変更を混ぜない
- 不要な抽象化を増やさない
- 将来使うかもしれないだけの抽象化を作らない

リファクタリングが必要な場合は、先に理由を説明する。

指示されていない大規模リファクタリングを勝手に行わない。

---

## 作業ログ・報告書・Todo管理

AIは、作業のたびに作業報告書を作成・更新する。

作業報告書とTodoリストは、次回以降の作業で過去の経緯を素早く把握し、不要な再調査やコンテキスト消費を減らすために使う。

作業内容を会話だけに残さず、プロジェクト内のドキュメントとして残すことを優先する。

---

## 作業報告書

作業ごとに、作業報告書を作成または追記する。

作業報告書には、必ず作業日時を明記する。

日時は以下の形式で記録する。

- YYYY年MM月DD日 HH時MM分SS秒
- 例: 2026年06月25日 12時34分56秒

日時は推測で書かない。

可能であれば、作業環境の現在時刻を取得して正確に記録する。

現在時刻を取得できない場合は、その旨を明記する。

---

## 作業報告書に必ず含める内容

作業報告書には、以下を必ず含める。

作業報告書を読めば、次回の作業者が会話ログをすべて読み直さなくても、作業の経緯と次にやるべきことを理解できる状態にする。

---

## 作業報告書の保存場所

作業報告書は、原則として以下のいずれかに保存する。

- docs/reports/
- docs/work_logs/
- reports/
- work_logs/

プロジェクトに既存のドキュメント管理方針がある場合は、それに従う。

保存場所が存在しない場合は、作成してよい。

ファイル名は、日付と内容が分かる名前にする。

例。

- 2026-06-25-ci-setup-report.md
- 2026-06-25-task-model-report.md
- 2026-06-25-frontend-task-card-report.md

---

## Todoリスト管理

プロジェクトにはTodoリストを作成し、作業のたびに更新する。

Todoリストは、次に何をするべきかを明確にするために使う。

Todoリストには、以下を含める。

- 未着手のタスク
- 進行中のタスク
- 完了したタスク
- 保留中のタスク
- 技術的負債
- 後で確認すること
- 次回最初に着手するタスク

Todoリストは、原則として以下のいずれかに保存する。

- docs/TODO.md
- TODO.md
- docs/project-todo.md

既存のTodo管理ファイルがある場合は、それを優先して更新する。

---

## Todoリストの更新ルール

作業開始時に、Todoリストを確認する。

作業中に新しいタスクや懸念が見つかった場合は、Todoリストに追記する。

作業完了時に、Todoリストを更新する。

Todoリストでは、各タスクの状態を明確にする。

推奨する状態は以下。

- 未着手
- 進行中
- 完了
- 保留
- 要確認

完了したタスクは削除せず、完了として残す。

ただし、Todoリストが長くなりすぎる場合は、完了済みタスクを作業報告書へ移動して整理してよい。

---

## 次回作業のための引き継ぎ

作業報告書の最後には、必ず次回作業のための引き継ぎを書く。

引き継ぎには以下を含める。

- 次にやるべきこと
- 次回最初に読むべきファイル
- 次回最初に実行すべきコマンド
- 注意すべき設計判断
- 触らない方がよい箇所
- 未確定の判断
- 確認が必要な仕様

次回のAIまたは開発者が、最小限のコンテキストで作業を再開できる状態にする。

---

## トークン削減のための記録方針

作業報告書とTodoリストは、次回以降のコンテキスト削減を目的として作成する。

そのため、以下を意識する。

- 会話ログを読まなくても経緯が分かるようにする
- 変更理由を簡潔に残す
- 設計判断を明記する
- 次に見るべきファイルを明記する
- 次に実行すべきコマンドを明記する
- 未解決事項を曖昧にしない
- 不要に長い説明を書かない

報告書は長文にしすぎず、次回作業に必要な情報を優先する。

## 作業報告書テンプレート

作業報告書は、原則として以下の構成で書く。

# 作業報告書

## 作業日時

YYYY年MM月DD日 HH時MM分SS秒

## 作業対象

対象機能、対象ファイル、対象レイヤーなどを書く。

## 作業目的

今回の作業で達成したいことを書く。

## 変更内容

何をどう変更したのかを書く。

## 変更したファイル

変更したファイルを列挙する。

## 変更意図

なぜその変更をしたのかを書く。

## 設計上の意図

責務分離、依存方向、テスト容易性、保守性などの観点から意図を書く。

## 影響範囲

影響する機能、API、画面、DB、テストなどを書く。

## 追加・更新したテスト

追加または更新したテストを書く。

## 実行した確認コマンド

実行したコマンドと結果を書く。

## CIで確認される内容

CIで確認される項目を書く。

## 未解決の課題

まだ解決していない課題を書く。

## 次にやること

次回以降にやるべきことを書く。

## 次回最初に見るべきファイル

次回作業時に最初に確認するべきファイルを書く。

## 引き継ぎ事項

注意点、未確定事項、設計判断、触らない方がよい箇所などを書く。

---

## 実装後の必須更新

実装後は、コード変更だけで終わらせてはいけない。

必ず以下を更新する。

- 作業報告書
- Todoリスト
- 必要に応じてADR
- 必要に応じてREADME
- 必要に応じてAPI仕様書
- 必要に応じてDB設計メモ

作業報告書とTodoリストを更新していない場合は、作業完了扱いしてはいけない。

---

## 禁止事項

以下は禁止する。

- CIなしで機能実装を進める
- 指示されていない大規模リファクタリング
- 既存設計を無視した実装
- テストなしの機能追加
- any の安易な使用
- unwrap / expect の安易な使用
- エラー処理の省略
- 巨大コンポーネントの作成
- 巨大handlerの作成
- 1ファイルへの詰め込み
- 不要な依存ライブラリ追加
- UIとビジネスロジックの密結合
- DB構造とdomain modelの密結合
- 動作確認していないのに完了と言い切る
- formatterやlintを後回しにする
- CIで実行していない検証を確認済みと言う
- コメントでごまかして未実装を残す
- 秘密情報をコードやログに出す
- 認証・認可を省略する
- 本番運用を無視したDB変更を行う

---

## 実装後に必ず報告すること

実装後は以下を報告する。

1. 変更した内容
2. 変更したファイル
3. 追加・更新したテスト
4. 実行した確認コマンド
5. CIで確認される内容
6. 作成・更新した作業報告書
7. 更新したTodoリスト
8. 残っている懸念
9. 次にやるべきこと
10. 次回最初に見るべきファイル

テスト、lint、format、typecheck、buildを実行できなかった場合は、実行できなかった理由を明記する。

作業報告書とTodoリストを更新していない場合は、完了扱いしてはいけない。

---

## ユーザーの学習方針

コードを丸ごと置き換えるのではなく、可能な限り差分で示す。

なぜその変更が必要なのかを短く説明する。

初学者向けにしすぎず、実務寄りの設計意図も説明する。

ただし説明が長すぎる場合は、先に結論とコードを出す。

既存コードを尊重し、勝手に別アーキテクチャへ作り替えない。

## 作業時の最終確認

作業完了前に、以下を確認する。

- CIは存在するか
- type checkは通るか
- formatter checkは通るか
- lintは通るか
- unit testは通るか
- buildは通るか
- 1ファイル300行以内に収まっているか
- model / usecase / hook / component に必要なテストがあるか
- コミット単位が小さく分けられるか
- 長期的に保守しやすい設計になっているか
- セキュリティ上の懸念が確認されているか
- DB変更が安全に行えるか
- API変更の影響範囲が説明できるか
- ログ・エラー設計が適切か
- 残課題が明記されているか
