## 開発で重視していること

以下を重視する。

- 設計してから実装すること
- CI First
- type check / format / lint / test / build を必ず通すこと
- ユニットテストを書くこと
- 小さな差分で進めること
- 小さなコミット単位で進めること
- 1ファイルに責務を詰め込みすぎないこと
- 1ファイルは原則300行以内にすること
- 規模に合ったクリーンアーキテクチャを使うこと
- 技術選定の理由を長期目線で説明すること
- 既存コードや既存設計を尊重すること
- 作業報告書とTodoリストを更新すること

---

## 避けてほしいこと

以下は避けてほしい。

- いきなり実装すること
- テストなしで完了扱いすること
- CIなしで機能実装すること
- 既存設計を無視して勝手に作り替えること
- 関係ない大規模リファクタリングをすること
- 1ファイルに大量の責務を詰め込むこと
- handlerやcontrollerにビジネスロジックを書くこと
- フロントエンドで巨大コンポーネントを作ること
- any、unwrap、expectなどで雑に逃げること
- エラー処理を省略すること
- 依存ライブラリを理由なく追加すること
- 動作確認していないのに完了と言い切ること
- 説明だけ長くて実装が進まないこと

## アーキテクチャ方針

規模に応じて、過剰設計にならない範囲でクリーンアーキテクチャを使いたい。

重視する分離。

- domain
- application / usecase
- infrastructure
- presentation / handler
- repository interface
- DTO
- error handling

domain層には、DB、HTTP、frameworkの都合を入れたくない。

handlerやcontrollerは薄くしたい。

usecaseにはアプリケーション固有の処理を置きたい。

repositoryは永続化の抽象化として扱いたい。

---

## 設計判断の優先順位

設計や実装で迷った場合は、原則として以下の優先順位で判断する。

1. 長期保守性
2. テスト容易性
3. 型安全性
4. 責務分離
5. 実装のシンプルさ
6. 既存設計との整合性
7. パフォーマンス
8. 開発速度
9. 学習効果

ただし、プロトタイプや検証目的の場合は、開発速度を一時的に優先してよい。

その場合でも、後から作り直せるように責務を分け、技術的負債を作業報告書やTodoリストに残す。

### バックエンド

- model追加 + model test

## コードレビュー観点

AIにコードレビューを依頼した場合は、以下の観点で確認してほしい。

- 責務が混ざっていないか
- テストしにくい構造になっていないか
- 既存設計を壊していないか
- 将来拡張するときに詰まらないか
- APIやDB変更に破壊的影響がないか
- セキュリティ上危険な実装がないか
- 1ファイルに詰め込みすぎていないか
- any、unwrap、expectなどで雑に逃げていないか
- エラー処理が適切か
- ログに秘密情報や個人情報が出ていないか
- handlerやcontrollerが太くなっていないか
- domainにDBやHTTPの都合が混ざっていないか
- CIで検証できる状態になっているか
- テストが正常系だけでなく異常系も含んでいるか

## 就活・キャリアの前提

現在の就活軸は、無理に特定の業界やプロダクトへの強い思い入れを作ることではなく、以下の4つを重視している。

1. 成長環境
2. 開発レベル
3. 将来性
4. 待遇

価値提供そのものは、企業である以上ある程度前提だと考えている。

そのうえで、自分にとって企業選びの差分は以下にある。

- どれだけ成長できるか
- どれだけ高いレベルの開発に触れられるか
- 今後も伸びる事業や市場か
- 給与や福利厚生を含めて長く働く上で納得感があるか

## 就活軸の詳細

### 成長環境

重視する観点。

- 若手でも手を動かせるか
- 実装だけでなく設計や課題整理にも触れられるか
- 優秀な人の近くで学べるか
- フィードバックやレビューの文化があるか

### 開発レベル

重視する観点。

- 技術品質を軽視していないか
- 設計、レビュー、テストの文化があるか
- エンジニアがプロダクトや価値に関与できるか
- 既存の仕組みや技術的負債にどう向き合っているか

### 将来性

重視する観点。

- 市場が今後も伸びるか
- その会社に競争力や独自性があるか
- エンジニアに継続して投資される環境か
- 今後のキャリアの選択肢が広がるか

### 待遇

重視する観点。

- 給与水準が納得できるか
- 福利厚生が整っているか
- 働き方や制度に無理がないか
- 長く働くうえで安心できる条件か

## なぜエンジニアか

エンジニアを志望する一番大きな理由は、純粋に作ることが楽しいから。

考えたものを、自分の手で実際に動く形にできることに面白さを感じている。

もう一つの理由は、直接接客するよりも、多くの人に価値を届けやすいから。

エンジニアは、対面で一人ひとりに価値を届ける仕事とは違い、一度作ったものを通して、より多くの人に価値を届けられる可能性がある。

まとめると、エンジニアを志望する理由は以下の3つ。

- 作ること自体が楽しい
- 技術で多くの人に価値を届けやすい
- その上で、成長や待遇のリターンも大きい
